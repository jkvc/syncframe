# Syncframe Kickoff — Design Discussion Summary

**Date:** 2026-06-06

## The Core Insight

Two browsers cannot agree on "where we are in a continuous state" by exchanging position messages — network jitter makes them diverge. But they **can** agree on a deterministic function that computes state-as-of-time, given a shared clock.

So instead of streaming state, we broadcast **anchors**: a server timestamp + a value at that moment + a motion descriptor. Any client with a synced clock can compute the current value at any later moment via a pure function. Anchors are rare events (only when motion changes — typically minutes apart), so broadcast cost is negligible.

This is **dead reckoning** from multiplayer game networking, generalized for arbitrary continuous state and exposed through a browser-native API.

## Two-Layer Architecture

### Layer 1: `@syncframe/core` — Time-Event Protocol

The pure sync primitives. No notion of screens, poses, or spatial layout. Just time.

**Provides:**
- NTP-style clock offset estimation (client-side)
- `Anchor<T, M>` — deterministic state at a point in time
- Evaluators — pure functions: `evaluate(anchor, serverNowMs) → T`
- Built-in motion shapes: `constant`, `linear1d`, `linear2d`, `bouncing`, `playback`
- Smoothed value chase — exponential lerp that absorbs network jitter
- `SyncStore` + `SyncTransport` interfaces (pluggable backends)

**Use cases it enables alone:**
- Synchronized video playback (watch parties)
- Synchronized audio (multi-device speaker arrays)
- Synchronized countdowns / stage timers
- Any continuous state that evolves over time

**Zero runtime dependencies.** Ships with `InMemoryStore` + `EventEmitterTransport` by default.

---

### Layer 2: `@syncframe/spatial` — World + Screens

Screen registry, poses, calibration UI. Builds on Layer 1 for use cases where screens have a geometric position.

**Provides:**
- `Screen` — named endpoint with a `ScreenPose` (world-coordinate bbox)
- `ScreenPose` — `{ worldX, worldY, worldWidth, worldHeight, rotationDeg }`
- Screen registration, heartbeat (presence), session management
- Calibration UI (React components: `CalibrationGrid`, `TopDownRoomMap`, `PoseEditor`)
- `ChromeFreeDisplay` — fullscreen immersive display surface

**Use cases it enables:**
- Multi-screen installations (panoramic image across N monitors)
- Distributed 2D canvas (multiple tablets form one drawing surface)
- Spatial audio (N phones play different channels of one piece)
- Any content that needs to know "what slice of the world does this screen render?"

## Storage Backends

The library is backend-agnostic. Multiple implementations:

### In-Memory (built into core)
- `InMemoryStore` — JS Map, perfect for single-process demos
- `EventEmitterTransport` — Node.js EventEmitter for pub/sub
- **Use case:** Local dev, hackathons, Cloudflare Workers
- **Limitation:** Single process, state lost on restart

### Redis
- `RedisStore` — JSON strings under `syncframe:*` keys
- `RedisPubSub` — Redis pub/sub for transport
- **Use case:** Multi-worker deployments, persistent state
- **Limitation:** Requires Redis

### Postgres
- `PostgresStore` — `syncframe_anchors`, `syncframe_screens` tables
- `PostgresTransport` — LISTEN/NOTIFY for pub/sub
- **Use case:** Postgres-only deployments
- **Limitation:** Slower than Redis, requires Postgres 12+

## Framework Adapters

Thin adapters that wire framework-specific request shapes to the core library:

- `@syncframe/adapter-next` — Next.js API route handlers
- `@syncframe/adapter-express` — Express route handlers
- `@syncframe/adapter-hono` — Hono route handlers

Each adapter is ~30 LOC of request-shape translation.

## Why "Syncframe"?

**Name rationale:**
- **"Sync"** — what it does (synchronizes continuous state)
- **"Frame"** — double meaning:
  - A **video frame** (time) — Layer 1 syncs video playback
  - A **display frame** (spatial) — Layer 2 manages screen geometry

The name grows with the library. Layer 1 syncs the *time-frame*, Layer 2 manages *screen-frames* in space.

**Alternatives considered:**
- `multireckon` — available but generic prefix
- `deadreckon` — distinctive but harder to google
- `syncframe` — clean, professional, double-meaning ✅
