# Syncframe Kickoff ‚Äî Design Discussion Summary

**Date:** 2026-06-06

> **Historical record.** This is the original design discussion. Some specifics here were aspirational and have since been refined ‚Äî see [`2026-06-06-package-architecture.md`](./2026-06-06-package-architecture.md) for the package design as actually built (core `/server` + `/react` entry points, the standalone `@syncframe/redis` adapter). Backend/adapter names below marked *(planned)* do not exist yet.

## The Core Insight

Two browsers cannot agree on "where we are in a continuous state" by exchanging position messages ‚Äî network jitter makes them diverge. But they **can** agree on a deterministic function that computes state-as-of-time, given a shared clock.

So instead of streaming state, we broadcast **anchors**: a server timestamp + a value at that moment + a motion descriptor. Any client with a synced clock can compute the current value at any later moment via a pure function. Anchors are rare events (only when motion changes ‚Äî typically minutes apart), so broadcast cost is negligible.

This is **dead reckoning** from multiplayer game networking, generalized for arbitrary continuous state and exposed through a browser-native API.

## Two-Layer Architecture

### Layer 1: `@syncframe/core` ‚Äî Time-Event Protocol

The pure sync primitives. No notion of screens, poses, or spatial layout. Just time.

**Provides:**
- NTP-style clock offset estimation (client-side)
- `Anchor<T, M>` ‚Äî deterministic state at a point in time
- Evaluators ‚Äî pure functions: `evaluate(anchor, serverNowMs) ‚Üí T`
- One built-in motion shape: `ScalarMotion` (a number advancing at a constant `ratePerMs`). Other shapes (2D paths, bouncing, playback, color lerp) are consumer-defined ‚Äî core stores them as opaque JSON
- Smoothed value chase ‚Äî exponential lerp that absorbs network jitter
- `SyncStore` + `SyncTransport` interfaces (pluggable backends)

**Use cases it enables alone:**
- Synchronized video playback (watch parties)
- Synchronized audio (multi-device speaker arrays)
- Synchronized countdowns / stage timers
- Any continuous state that evolves over time

**Zero runtime dependencies.** Ships with `InMemoryStore` + `EventEmitterTransport` by default.

---

### Layer 2: `@syncframe/spatial` ‚Äî World + Screens

Screen registry, poses, calibration UI. Builds on Layer 1 for use cases where screens have a geometric position.

**Provides:**
- `Screen` ‚Äî named endpoint with a `ScreenPose` (world-coordinate bbox)
- `ScreenPose` ‚Äî `{ worldX, worldY, worldWidth, worldHeight, rotationDeg }`
- Screen registration, heartbeat (presence), session management
- Calibration UI (React components: `CalibrationGrid`, `TopDownRoomMap`, `PoseEditor`)
- `ChromeFreeDisplay` ‚Äî fullscreen immersive display surface

**Use cases it enables:**
- Multi-screen installations (panoramic image across N monitors)
- Distributed 2D canvas (multiple tablets form one drawing surface)
- Spatial audio (N phones play different channels of one piece)
- Any content that needs to know "what slice of the world does this screen render?"

## Storage Backends

The library is backend-agnostic. Multiple implementations:

### In-Memory (built into core)
- `InMemoryStore` ‚Äî JS Map, perfect for single-process demos
- `EventEmitterTransport` ‚Äî Node.js EventEmitter for pub/sub
- **Use case:** Local dev, hackathons, Cloudflare Workers
- **Limitation:** Single process, state lost on restart

### Redis ‚Äî shipped as `@syncframe/redis`
- `RedisStore` ‚Äî JSON strings under `syncframe:*` keys
- `RedisTransport` ‚Äî Redis pub/sub for transport
- Connection-injected (caller supplies `ioredis` clients); type-only dependency on core
- **Use case:** Multi-worker deployments, persistent state
- **Limitation:** Requires Redis

### Postgres *(planned)*
- `PostgresStore` ‚Äî `syncframe_anchors`, `syncframe_screens` tables
- `PostgresTransport` ‚Äî LISTEN/NOTIFY for pub/sub
- **Use case:** Postgres-only deployments
- **Limitation:** Slower than Redis, requires Postgres 12+

## Framework Adapters *(planned)*

Thin adapters that wire framework-specific request shapes to the core library:

- `@syncframe/adapter-next` ‚Äî Next.js API route handlers
- `@syncframe/adapter-express` ‚Äî Express route handlers
- `@syncframe/adapter-hono` ‚Äî Hono route handlers

Each adapter is ~30 LOC of request-shape translation. (Today the demo site wires routes directly against `@syncframe/core/server`; no adapter package exists yet.)

## Why "Syncframe"?

**Name rationale:**
- **"Sync"** ‚Äî what it does (synchronizes continuous state)
- **"Frame"** ‚Äî double meaning:
  - A **video frame** (time) ‚Äî Layer 1 syncs video playback
  - A **display frame** (spatial) ‚Äî Layer 2 manages screen geometry

The name grows with the library. Layer 1 syncs the *time-frame*, Layer 2 manages *screen-frames* in space.

**Alternatives considered:**
- `multireckon` ‚Äî available but generic prefix
- `deadreckon` ‚Äî distinctive but harder to google
- `syncframe` ‚Äî clean, professional, double-meaning ‚úÖ
